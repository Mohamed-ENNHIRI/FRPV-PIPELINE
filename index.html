Research presentation

National PV Roof Database

Building, from open public data, the first uniform inventory of every
photovoltaic panel on French rooftops — automatically, at national scale.

Research aim

Today, France has no single, geo-referenced source of truth describing where rooftop PV is installed, how each panel is oriented, and how much surface it covers. Existing registries (ENEDIS, BDPV, Open Power System Data) are either address-level, declarative, or incomplete — none of them tells a researcher, for a given roof, the panel's exact position, tilt and azimuth. This work fills that gap.

①   Open dataset

A reproducible, per-panel database covering all 96 metropolitan départements — position · tilt · azimuth · area · provenance.

②   Tractable pipeline

A fully open-source, single-machine pipeline that turns ~90 TB of public LiDAR into a queryable database in 3–5 days.

③   Research enablement

Plugs directly into solar-yield modelling, grid-impact studies, policy evaluation and self-consumption analysis — the kind of question that needs panel-level geometry.

Inputs
  • IGN LiDAR HD (national 3D point cloud)
  • Cadastre Etalab (building footprints)
  • BDOrtho 20 cm aerial imagery
  • PV-presence model scores
3-stage pipeline
  • Index  decide which tiles to fetch
  • Segment  extract roof planes (tilt / az.)
  • Assign  bind detected panels to planes
Output (per panel)
  • lon / lat (WGS-84), aream²
  • tilt & azimuth (deg)
  • building_id, source_tile, source_image

§ 3 · EXTRACTION METHODOLOGY 04 / 09

From 3D point cloud to georeferenced roof planes, in 7 steps.

Each LiDAR tile passes through the same deterministic algorithm. Below: every step, its key formula, and a schematic of what it does.

1
FILTER
Semantic filtration
$\mathcal{B} = \{\,p \in \mathcal{P} : \mathrm{class}(p) = 6\,\}$
2
VOXELIZE
Voxel-based hashing
$C_{q_x,q_y} = \arg\max_{p}\, Z(p)$
3
TRIANGULATE
Constrained Delaunay
$L \le \sqrt{4.5}\,\mathrm{m},\;|\Delta z| < 1.5\,\mathrm{m}$
4
GROW
Region growing
$\vec{n}_i\!\cdot\!\vec{n}_j \ge \cos 8°$
5
MERGE
Topological rectification
$\Delta\theta_{adj} \le 3°$
6
REPROJECT
CRS reprojection
$f:(x,y)_{2154}\!\to\!(\lambda,\phi)_{4326}$
7
EMIT
Metadata serialisation
$\beta = (90 - \arctan_2(n_y, n_x))\bmod 360°$

Output: one JSON per LiDAR tile · per roof region: tilt, azimuth, area, vertex polygon (WGS-84)

§ 4 · ROOF SEGMENTATION 05 / 09

From a 3D point cloud to roof planes.

For every PV-positive building in a tile, the engine isolates the roof points, fits flat regions, and emits one tilt/azimuth/area record per region — all in a JSON file per tile.

1Clip building points to cadastre
2Voxelize on a 1 m grid
3Triangulate & filter slivers
4Region-grow by surface normal
5Merge near-coplanar regions
6Project onto mean plane → JSON

§ 6 · COMPANION EXPERIMENT 07 / 09

From panel surface to installed capacity — validated against the national registry.

Each detected panel surface S is converted to capacity by a single rule: C = S / 5 [kWp] (5 m²/kWp baseline for mainstream c-Si modules). We sum per commune and compare against the official Registre National des Installations. Across 161 communes in dépt 82 the slope-through-origin is β = 0.964 — the imagery pipeline recovers ≈ 96 % of the registered rooftop capacity.

Figure 1 · per-commune validation · n = 161
each dot = one commune  ·  hover for details  ·  grey dashed = y = x  ·  navy = β × x
Filters & rules applied
Per-panel: capacity from area, $C = S / 5$ kWp (Fraunhofer ISE 2023 baseline).
Registry: codeFiliere = SOLAI, dépt 82, snapshot 31-12-2022.
Rooftop subset: aggregated rows of installations < 36 kW + individual installations ≤ 500 kW.
500 kW threshold: separates rooftop from ground-mount utility farms cleanly (95th pct = 915 kW vs the 1 MW class) — removes 40 rows totalling 139 MWp.
Commune key: INSEE code from building_id prefix (BD TOPO encoding).
